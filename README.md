# ME/CFS Research Pipeline — Hermes Agent Profile

An automated biomedical research pipeline that ingests PubMed articles, generates plain-language summaries, tracks treatment evidence, drafts editorial science newsletters, and exports ready-to-publish HTML — all running as a cron-driven Hermes profile.

## The REPORTS System

The pipeline is **multi-topic**. It rotates between three report types on a daily cadence:

| REPORT_TYPE | Condition | PubMed Search Query |
|---|---|---|
| `MECFS` | ME/CFS | `(ME/CFS OR myalgic encephalomyelitis OR chronic fatigue syndrome)` |
| `MCAS` | Mast Cell Activation Syndrome | `(MCAS OR mast cell activation syndrome OR mastocytosis)` |
| `HYPER` | Hypermobility / EDS | `(hypermobility OR hypermobile OR joint instability)` |

### REPORT.ini

The central configuration file (`workspace/REPORT.ini`) controls which report type is active and what date range to scan:

```ini
REPORT_TYPE=HYPER
REPORT_MECFS_DATE_FROM=2026-05-20
REPORT_MECFS_DATE_TO=2026-05-27
REPORT_MCAS_DATE_FROM=2026-05-20
REPORT_MCAS_DATE_TO=2026-05-27
REPORT_HYPER_DATE_FROM=2026-05-20
REPORT_HYPER_DATE_TO=2026-05-27
```

Each report type tracks its own date range independently, so catching up on a missed rotation is seamless.

## Pipeline: Step by Step

All jobs run daily starting at **11:05 UTC**. Each step reads from `REPORT.ini` to determine which `REPORT_TYPE` and date range to process.

### Step 0 — Refresh Report Type & Dates (11:05 UTC)
**Type:** No-agent script (`report-refresh.sh`)

Rotates `REPORT_TYPE` through the cycle (MECFS → MCAS → HYPER → MECFS ...) and updates the date ranges so each type's `_DATE_FROM` picks up where the previous run left off. This ensures no gaps or overlaps.

### Step 1 — Ingest Articles (11:08 UTC)
**Skill:** `research-roundup` (deepseek-v4-flash)

Queries PubMed with the `REPORT_TYPE`'s mapped search query for the current date range. Fetches metadata and full text (via PMC), partitions articles into `articles-{REPORT_TYPE}/{pub_date}/` by their actual publication date, and logs PMIDs to `processed.txt` to avoid re-downloading.

### Step 2 — Process & Summarize (11:13 UTC)
**Skill:** `research-processor` (deepseek-v4-flash)

Reads all unprocessed articles in the date range, skips already-summarized ones, and generates:
1. **Plain-language summaries** (`{PMID}-summary.md`) written for a college-educated lay audience
2. **Treatment profiles** (`treatment-{REPORT_TYPE}/{treatment}.md`) — tracks evidence levels, mechanisms, and clinical status for each therapy discussed

### Step 3 — Draft Blog Post (11:18 UTC)
**Skill:** `research-post-writer` (gpt-5.5 via openai-codex)

Aggregates all `*-summary.md` files from the date range, identifies biological themes and intersections between studies, and writes an editorial-style newsletter draft saved to `blog/{REPORT_TYPE}-post-YYYY-MM-DD.md`.

### Step 4 — Generate Feature Image (11:23 UTC)
**Skill:** `research-post-image` (deepseek-v4-flash)

Reads the blog post, crafts a visual prompt from its title and themes, generates a 1792x1024 PNG via OpenAI's `gpt-image-2`, and saves it alongside the markdown file.

### Step 5 — Export HTML (11:23 UTC)
**Skill:** `research-post-publish` (deepseek-v4-flash)

Reads the blog post markdown file, extracts the title and body, converts the body markdown to clean HTML via `scripts/convert_md_to_html.py`, and saves a complete HTML document alongside the markdown source. The HTML file is ready for copying into any publishing platform's editor.

## Required API Keys

The `.env` file must be configured with:

| Variable | Purpose |
|---|---|
| `DEEPSEEK_API_KEY` | LLM provider for Steps 1, 2, 4, 5 |
| `OPEN_AI_API_KEY` | Image generation (gpt-image-2) for Step 4 |
| `DISCORD_BOT_TOKEN` | Output delivery channel |

All other API keys in `.env` are optional or for other Hermes features (browser, web search, STT/TTS, messaging platforms).

## Setting Up a Fresh Copy

1. **Copy the profile** to a new location:
   ```bash
   cp -r /opt/data/profiles/med-research /opt/data/profiles/my-research
   ```

2. **Configure `.env`** with your own API keys

3. **Register cron jobs** (see `cron/jobs.json` for reference):
   ```bash
   # Each job needs profile and workdir set:
   cronjob action=create profile=my-research workdir=/opt/data/profiles/my-research ...
   ```

4. **Verify the pipeline** by running Step 0 manually:
   ```bash
   cronjob action=run job_id=<step0-job-id>
   ```

## Efficiency Notes

All pipeline skills are optimized for cron execution:

- **No memory/session_search calls** — variables come from the cron prompt and `REPORT.ini`
- **No `todo` tool** — track progress in reasoning only
- **Batch file reads** — use `execute_code` with `read_file` loops to combine I/O
- **No post-write verification** — `write_file` return confirms success
- **No re-processing** — already-summarized articles and processed PMIDs are skipped
- **Scripted collection** — `research-post-writer` uses `scripts/collect_summaries.py` for batch reads

The `cron-session-optimizer` meta-skill analyzes past runs and patches inefficiencies automatically.

## Cron Schedule (Daily UTC)

```
11:05   Step 0  ─► Report Type & Date Refresh
11:08   Step 1  ─► PubMed Ingestion
11:13   Step 2  ─► Process & Summarize
11:18   Step 3  ─► Draft Blog Post
11:23   Step 4  ─► Generate Feature Image
11:23   Step 5  ─► Export HTML
```

Steps 4 and 5 share a schedule slot and run in parallel. The pipeline touches three report types in rotation, so each type gets processed approximately every 3 days.
